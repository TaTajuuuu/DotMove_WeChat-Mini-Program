# Project Constitution

这是一点运动微信运动打卡小程序 1.0.0 的开发原则

- 项目名称：一点运动微信打卡小程序  
- 版本号：v1.0.5  
- 创建日期：2026-06-08  
- 最后更新：2026-06-22  
- 负责人：獭獭  
- 适用范围：本项目的需求分析、产品设计、开发、测试、上线与维护全过程

## 1. Project Goal

- 本项目旨在为运动社群提供目标设定、打卡、统计复盘、自我监督及成员间互相激励功能，降低人工统计成本，提高用户持续参与率。
- 核心解决社群成员的运动情况记录及展示需求。
- 服务于微信运动群的社群成员。
- 不追求复杂社交功能，不做泛健身内容平台。
- 聚焦小规模社群的运动行为记录与反馈。
- 成功标准：
 - 能满足成员目标设定、打卡记录、统计复盘的需要，创建者能在 1 分钟内创建小组并邀请成员；成员能在 3 分钟内完成月计划设定，在非首次授权且网络与图片上传正常的情况下，30 秒内完成一次打卡。
 - 能有效起到自我监督及成员间互相激励的作用，支撑成员坚持运动习惯及保持参与状态。
 - 创建者不再需要手工统计每日、每月完成情况，有效降低小组创建者人工统计及管理成本。

## 2. MVP Scope / v1.0范围边界

本项目当前阶段只聚焦以下功能：

  1. 创建运动小组：创建者可选择创建本月小组或下月小组，每个小组人数上限50人
  2. 小组生命周期：默认小组生命周期为一个自然月，即每月第一天到最后一天为小组打卡功能激活时间。生命周期开始前，除打卡和补卡外其他功能可用；生命周期结束后进入归档，仅可只读查看。如本月小组在月中创建，生命周期仍从当月 1 日到当月最后一天，成员加入前的日期允许补卡昨天和前天，其余日期按未打卡进行统计。
  3. 创建者可自定义、修改小组名称
  4. 加入运动小组：可通过微信群分享卡片、邀请码，加入不需要创建者审核。
  5. 自定义个人在小组内的昵称
  6. 设置个人运动目标类型，运动目标类型有：运动天数、运动次数、月总运动时长、月总最低消耗热量、跑步距离、骑行距离、三环闭合天数。
  7. 设置个人运动目标值：成员先选择运动目标类型（可选多项），再对所选的目标类型进行目标数值设定。运动天数-天（单天运动最低消耗热量-kcal），运动次数-次（单次运动最低消耗热量-kcal），月总运动时长-小时，月总最低消耗热量-kcal。跑步距离-km，骑行距离-km，三环闭合天数-天。目标配置状态按“成员-小组-自然月”的整组目标配置管理，不按单个目标类型分别管理。
  8. 设置月度目标对应“一点币”币值，“一点币”仅为小程序内部计量单位。只是用户给月度目标设置的虚拟承诺值/计量值，与月度目标一起进行展示。
  9. 目标周期为：月度目标。
  10. 运动打卡：用户根据已设置的目标类型，手动填写当天实际运动数值，如：时长、热量、距离、是否三环闭合等；提交运动照片（至少一张、最多三张）及运动状态备注（选填，最多 100 字）；同一成员、同一小组、同一运动日期的有效打卡总数不得超过 5 次，打卡可于当天修改运动数值、增删运动照片及修改备注。
  11. 运动补卡：可以补昨日或前天的卡，每个用户每个自然日最多发起 3 次补卡（如6月1日运动四次未打卡，6月2日可补6月1日三次运动打卡）。补卡需填写内容与打卡一样，补卡记录计入同一成员、同一小组、被补运动日期的 5 次有效打卡上限。补卡可于提交当天修改运动数值、增删运动照片及修改备注；补卡无需填写原因、无需审批，小组内有效成员可见补卡标记及内容。
  12. 查看打卡情况
  13. 小组成员可见： 小组内有效成员可以看到其他成员的昵称、目标、打卡状态、打卡/补卡运动照片及备注、统计结果。通过“互相可见打卡状态及目标完成情况”产生成员间互相激励的效果。
 
  14. 统计及复盘：统计每一类目标类型的完成进度，统计已完成月度目标的成员及完成日期，统计小组整体完成率。小组归档时，应按归档时刻冻结统计快照并只读展示。
  15. 成员退出小组：active 成员可主动退出小组，退出后普通页面不可见且不计入当前统计。
  16. 创建者移除成员：创建者可移除 active 成员，被移除成员在 v1.0.0 MVP 中不可重新加入同一小组。
  17. 创建者转让与退出：创建者可将创建者身份转让给其他 active 成员；创建者退出自己创建的小组前必须先完成创建者身份转让。
  18. 创建者解散小组：创建者可解散小组，解散后创建者和成员均不可在普通页面查看该小组，仅后台审计或争议追溯可查。


当前版本明确不包含以下功能：

  1. 复杂社交（如：允许点赞、评论、提醒对方打卡等）
  2. 排行榜
  3. 商城
  4. AI推荐
  5. 运动建议
  6. 动作指导
  7. 接入微信运动步数
  8. 通知功能（如：订阅消息、打卡提醒、补卡提醒、小组活动开始/结束提醒等）
  9. 一点币结算、奖惩、统计、充值及目标完成返还或未完成扣除功能
  10. 创建者设置小组“可搜索/不可搜索”
  11. 小程序内搜索公开小组。
 


## 3. Core Principles / 核心开发原则

### 3.1 用户价值优先

所有功能必须服务于明确的用户场景。<br>  
不得为了技术展示、界面炫技或功能堆砌而增加复杂度。

判断标准：
- 是否解决真实用户问题？
- 是否降低用户操作成本？
- 是否提高关键业务指标？

### 3.2 简单优先

在满足需求的前提下，优先选择最简单、最稳定、最容易维护的方案。

要求：
- 优先使用成熟方案，而非过度设计。
- 优先保证核心流程闭环，而非一次性覆盖所有边缘场景。
- 新增复杂机制前，必须说明其必要性。
- 小程序主包体积必须控制在 2MB 以内。
- 本地不得放置大量图片、视频、字体、测试文件。
- 所有功能必须先有需求说明、验收标准，再开发代码。
- 每次只开发一个明确任务，完成后必须测试。


### 3.3 数据可追踪

关键业务行为必须可记录、可统计、可分析。

包括但不限于：
- 创建小组
- 加入小组
- 关键功能使用
- 表单提交
- 状态变化
- 错误日志
- 运营指标

### 3.4 稳定性优先于功能扩张

任何新功能不得破坏已有核心流程。

上线前必须确保：
- 主流程可用
- 数据不丢失
- 异常有提示
- 失败可恢复

### 3.5 可维护性优先

代码、数据结构、接口和文档必须便于后续维护。

要求：
- 命名清晰
- 模块职责明确
- 避免重复代码
- 关键逻辑必须有注释
- 重要决策必须有记录

## 4. User experience Constraints / 用户体验约束

所有交互设计必须遵守以下原则：

- 核心操作路径不超过 3 步。
- 用户提交重要信息前必须有明确提示。
- 操作成功、失败、加载中必须有状态反馈。
- 空状态、错误状态、无权限状态必须有说明。
- 移动端优先

## 5. access right constraint / 权限约束

系统至少包含以下角色：

| 角色 | 用户描述 | 核心诉求 | 权限范围 | 不可操作内容 |
|---|---|---|---|---|
| 小组创建者 | 发起运动打卡机制的人 | 创建小组并要求朋友加入 | 创建\分享小组、查看成员目标和统计、移除成员、转让创建者、退出或解散小组 | 修改\移除其他成员的目标、打卡记录；移除成员后，该成员历史数据物理保留，但对普通页面不可见，且不计入当前小组统计。创建者退出自己创建的小组前，必须先将创建者身份转让给其他 active 成员。 |
| 小组成员 | 微信群里的朋友 | 设定个人目标并坚持打卡 | 加入小组、退出小组、设置目标、运动打卡、查看个人目标、打卡记录及复盘统计信息、查看其他成员的目标、打卡记录及复盘统计信息 | 修改\移除其他成员的目标、打卡记录 |
| 管理员 | 后续版本可能需要的维护角色 | 处理成员、数据异常和规则调整 | 暂时不做这个角色的功能 | 暂时不做这个角色的功能 |

权限原则：
- 用户只能访问自己有权限的数据。
- 小组创建者\管理员的操作必须可追踪。
- 敏感操作必须二次确认。
- 退出成员允许通过原邀请码或分享卡片重新加入同一小组；重新加入后恢复 active 成员关系，退出前及退出期间的数据均不补回当前统计。重新加入不重置该成员在该小组该自然月的目标配置状态；如目标配置已 locked，仍不可修改；如仍为 unset，则按当前小组状态规则处理首次设置。
- 被创建者移除的成员在 v1.0.0 MVP 中不允许重新加入同一小组；如后续版本需要恢复加入能力，必须重新评审成员关系、审计和统计口径。

## 6. Technical Constraints / 技术约束

### 6.1 技术栈约束

- 前端：微信小程序原生框架  
- 后端：微信云开发  
- 数据库：微信云开发数据库  
- 部署环境：微信小程序平台 + 微信云开发环境  
- 第三方服务：微信登录、云存储；订阅消息为后续版本预留，v1.0 不启用。
- 原则：
 - 未经评估，不引入新的核心技术栈。
 - 第三方依赖必须说明用途、风险和替代方案。
 - 不使用停止维护或安全风险较高的依赖。

### 6.2 架构约束

系统架构必须遵守以下原则：

- 前端、后端、数据库职责清晰。
- 业务逻辑不得过度耦合在页面层。
- 核心业务逻辑应具备可复用性。
- 配置项不得硬编码在业务代码中。
- 关键模块之间必须通过清晰接口通信。

### 6.3 接口约束

所有 API 必须遵守以下规范：

- 请求参数必须校验。
- 响应结构必须统一。
- 错误码必须可识别。
- 接口必须考虑权限控制。
- 重要接口必须记录日志。

标准响应格式：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {}
}
```

## 7. data constraint / 数据约束

### 7.1 数据设计原则

数据库设计必须遵守：

- 表名、字段名语义清晰。
- 关键字段必须设置索引。
- 重要数据必须记录创建时间和更新时间。
- 不允许随意删除核心业务数据。用户退出、移除成员或解散小组后，该成员/小组历史数据（名称、昵称、打卡/补卡数据、运动照片及备注）物理保留，但对普通页面不可见，且不计入统计；小组解散后，创建者和成员均不可在普通页面查看该小组，仅后台审计或争议追溯可查。
- 状态字段必须定义清楚枚举值。
- 不要把用户上传图片、背景大图、演示视频、无关截图、未压缩素材直接放进 项目目录。
- 图片类资源优先使用云存储或远程 CDN。
- 打卡和补卡上传的运动照片必须使用云存储，不得放入小程序本地代码包。

### 7.2 数据安全原则

涉及用户数据时，必须遵守：

- 最小化收集原则。
- 不存储无必要的敏感信息。
- 敏感数据不得明文暴露。
- 用户数据导出、删除、修改必须有权限控制。
- 重要操作必须保留审计记录。

## 8. Coding Standards / 代码规范

### 8.1 命名规范

- 变量名必须表达业务含义。
- 函数名必须体现动作。
- 组件名必须体现用途。
- 禁止使用无意义命名，如 data1、temp、test2。

### 8.2 代码结构

代码必须按照模块组织：

```text
/program
  /components
  /pages
  /services
  /utils
  /types
  /config
```

要求：
- 页面层只负责展示和交互。
- service 层负责接口调用。
- utils 只放通用工具函数。
- 类型定义集中管理。

### 8.3 注释原则

必须写注释的情况：

- 复杂业务规则
- 特殊兼容逻辑
- 权限判断逻辑
- 数据转换逻辑
- 非直观算法

不需要注释的情况：

- 代码本身已经清楚表达含义
- 简单赋值
- 重复解释函数名

## 9. Testing Principles / 测试原则

### 9.1 测试方式

- 小程序端自测
- 云函数测试
- 数据库权限测试
- 核心流程验收清单

### 9.2 必测范围

以下内容必须测试：

- 创建小组
- 加入小组
- 各角色权限
- 目标类型及目标值设定
- 打卡
- 补卡
- 表单校验
- 数据新增、修改、删除
- 异常状态
- 边界条件
- 移动端适配

### 9.3 测试通过标准

上线前必须满足：

- 核心流程无阻断问题。
- 高优先级 Bug 清零。
- 关键页面无明显 UI 错乱。
- 关键接口响应正常。
- 数据写入、读取、更新准确。

## 10. SDD Workflow / SDD 工作流

### 10.1 完整工作流：从原则到上线

 0.	Project Principles / Constitution 项目原则 
 1.	PRD 产品需求文档
  1.1 PRD 写作计划
  2.1 PRD 大纲确认
  3.1 分章节生成正文
  4.1 问题清单确认
  5.1 PRD 定稿检查
 2.	Spec 功能规格说明
  2.1 确定 Spec 文档结构：
  建立 docs/02-spec.md 的目录骨架，明确本阶段只写“可开发、可测试的功能规格”，不写技术设计、不拆任务、不写代码。
  2.2 定义全局规则
  先写跨功能的底层口径：小组状态、成员关系状态、目标配置状态、打卡记录状态、时间口径、生命周期、归档只读、退出/移除/解散后的可见性。
  2.3 细化核心功能规格
  按 PRD 功能拆用户故事：创建小组、加入小组、昵称、目标设置、打卡、补卡、查看打卡、统计复盘、退出/移除/解散。
  2.4 补齐验收标准
  每个用户故事都写成“当……时，系统应……”格式，覆盖正常流程、失败流程、权限限制、边界条件。
  2.5 细化关键业务算法
  重点写清楚每种目标类型的进度计算、个人综合进度、月目标完成判定、完成日期、小组整体完成率、未设置目标配置状态。
  2.6 细化校验、提示和审计规则
  包括补卡 3 次校验、运动日期 5 次有效打卡上限、打卡/补卡修改范围、错误提示、空状态、无权限状态、隐私提示、加入确认。
  2.7 Spec 完整性复核与确认
  检查是否还有模糊词、规则冲突、验收缺口；确认后才能进入 03-technical-design.md 技术设计阶段。
 3.	Technical Design 技术设计方案
 4.	Tasks 开发任务拆解
 5.	Implement 逐任务编码
 6.	Test 验收与回归测试
 7.	Change Log 变更管理与规格回写

 |---|---|---|---|
 | 0 | 项目边界和开发原则 | 00-project-principles.md | 否 |
 | 1 | 产品为什么做、给谁用、做什么 | 01-prd.md | 否 |
 | 2 | 每个功能的规则和验收标准 | 02-spec.md | 否 |
 | 3 | 技术上如何实现 | 03-technical-design.md | 否 |

- 每一阶段都需要正式确认后再进入下一阶段。
- 每次只做一个阶段里的一个细分任务，只修改任务涉及文件。
- 如果需要修改额外文件，必须先说明原因。完成后必须列出改动文件和测试方法。

- 每次需求变更，先改 Spec，再改 Tasks，最后改代码。
- 过期文档不要直接删除，可以放入 archive，以便追溯。
- AI 编码工具每次执行前，必须明确它可以参考哪些文件、只能修改哪些文件。

## 11. Quality Gates / 质量门禁

任何需求进入开发前，必须满足：

- 需求目标明确。
- 用户场景明确。
- 验收标准明确。
- 页面 / 接口 / 数据影响范围明确。

任何功能进入上线前，必须满足：

- 已完成自测。
- 已完成核心流程测试。
- 已通过验收标准。
- 已更新相关文档。
- 已评估对既有功能的影响。

## 12. Change Management / 变更管理

### 12.1 需求变更原则

需求变更必须记录以下内容：

- 变更原因
- 变更内容
- 影响范围
- 优先级
- 是否影响上线时间
- 决策人

## 13. Definition of Done / 完成定义

一个功能只有同时满足以下条件，才可以视为完成：

1. 需求已实现。
2. 验收标准已通过。
3. 核心流程可正常运行。
4. 异常状态已处理。
5. 权限逻辑正确。
6. 数据记录正确。
7. UI 展示符合设计要求。
8. 代码已提交并通过检查。
9. 相关文档已更新。
10. 无阻断级 Bug。

## 14. Prohibited Practices / 禁止事项

本项目禁止以下行为：

- 未确认需求就直接开发。
- 为了赶进度跳过核心测试。
- 在业务代码中硬编码敏感信息。
- 未评估影响就修改数据库结构。
- 未记录原因就调整核心业务规则。
- 引入无必要的复杂架构。
- 为边缘场景牺牲主流程体验。
- 未经确认删除用户数据。

## 15. Priority Rules / 优先级规则

当发生冲突时，按照以下顺序决策：

1. 数据安全 > 功能完整
2. 核心流程稳定 > 新功能上线
3. 用户体验清晰 > 视觉复杂度
4. 可维护性 > 短期开发速度
5. 明确需求 > 主观猜测
6. 已验证方案 > 未验证创新
