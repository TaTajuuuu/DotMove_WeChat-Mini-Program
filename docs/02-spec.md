## 1. Spec 阶段说明

### 1.1 阶段目标

Spec 阶段用于把 PRD 中的产品语言转换为可开发、可测试、可验收的功能规则。

本阶段回答：

- 每个功能在什么条件下允许或禁止使用。
- 系统在正常、异常、边界、无权限状态下应如何响应。
- 关键业务对象有哪些状态，以及状态如何变化。
- 统计、校验、可见性、归档、数据保留等规则如何被验证。

本阶段不回答：

- 页面路径、组件拆分、数据库集合、云函数名称等技术实现问题。
- 具体开发任务拆分。
- UI 视觉设计细节。
- 代码实现。

- 每个功能必须包含用户故事。

## 2. 全局规则

### 2.1 业务对象定义

| 对象 | 定义 |
|---|---|
| 用户 | 使用一点运动微信打卡小程序的微信用户 |
| 小组 | 由创建者创建的月度运动打卡组织单元 |
| 创建者 | 创建小组并拥有小组管理权限的用户 |
| 成员 | 加入某个小组并参与目标设置、打卡、补卡和统计查看的用户 |
| 有效成员 | 成员关系为 active，且小组状态允许查看或参与统计的成员；文档中的“小组内有效成员”和“active 成员”按同一产品口径理解 |
| 成员关系 | 用户与某个小组之间的关系记录 |
| 月度目标 | 成员在某个小组、某个自然月内设置的运动目标 |
| 有效目标配置 | 目标配置状态为 set 或 locked，且所属成员在对应统计口径下可参与统计的目标配置 |
| 一点币 | 成员为月度目标设置的虚拟承诺值或计量值，仅展示，不参与结算、奖惩、统计、充值或扣除 |
| 打卡记录 | 成员提交的某个运动日期的实际运动数据 |
| 补卡记录 | 成员在当前自然日为昨日或前天补交的打卡记录 |
| 物理有效记录 | 物理保留且状态为 valid 或 edited 的打卡或补卡记录；无论是否计入当前统计，均计入同一成员、同一小组、同一运动日期的 5 次有效打卡上限 |
| 运动照片 | 打卡或补卡时用户上传的运动证明图片，每条记录至少 1 张、最多 3 张 |
| 备注运动状态 | 打卡或补卡时用户选填的文字备注，最多 100 字 |
| 运动日期 | 打卡记录实际归属的日期 |
| 提交日期 | 用户执行打卡或补卡提交操作的日期 |

### 2.2 时间口径

| 规则编号 | 规则 |
|---|---|
| TIME-001 | 系统中的自然日、昨日、前天、自然月均按北京时间计算。 |
| TIME-002 | 小组生命周期按自然月计算，从当月 1 日 00:00:00 到当月最后一日 23:59:59。 |
| TIME-003 | 本月小组即生命周期属于当前自然月的小组。 |
| TIME-004 | 下月小组即生命周期属于下一个自然月的小组。 |
| TIME-005 | 运动日期用于统计运动完成情况。 |
| TIME-006 | 提交日期用于校验当天打卡修改、补卡当天修改、每自然日补卡次数限制。 |
| TIME-007 | 当用户补卡时，运动日期必须是提交日期的昨日或前天。 |
| TIME-008 | 当本月小组在月中创建时，小组生命周期仍为当月 1 日至当月最后一日。 |
| TIME-009 | 当成员加入小组前存在未参与日期时，仅昨日和前天可按补卡规则补交，其余日期按未打卡统计。 |
| TIME-010 | 当 exited 成员重新加入同一小组后，如提交日期的昨日或前天属于退出期间，允许按补卡规则补交；退出期间的其他日期不得补卡。 |

### 2.3 小组状态

| 状态 | 含义 | 是否可打卡 | 是否可补卡 | 是否可设置或修改目标 | 是否可查看 |
|---|---|---|---|---|---|
| upcoming | 生命周期尚未开始 | 否 | 否 | 是 | 是 |
| active | 生命周期进行中 | 是 | 是 | 仅允许首次设置，已保存后不可修改 | 是 |
| archived | 生命周期自然结束后归档 | 否 | 否 | 否 | 是，仅归档快照生成时成员关系为 active 的成员可只读查看 |
| dissolved | 创建者主动解散 | 否 | 否 | 否 | 创建者和成员普通页面均不可见，仅后台审计或争议追溯可查 |

状态变化规则：

| 规则编号 | 当……时 | 系统应…… |
|---|---|---|
| GROUP-STATE-001 | 创建者创建下月小组成功时 | 将小组状态设为 upcoming。 |
| GROUP-STATE-002 | 创建者创建本月小组且当前日期位于该小组生命周期内时 | 将小组状态设为 active。 |
| GROUP-STATE-003 | upcoming 小组到达生命周期开始时间时 | 将小组状态变为 active。 |
| GROUP-STATE-004 | active 小组到达生命周期结束后的归档时间时 | 将小组状态变为 archived。 |
| GROUP-STATE-005 | 创建者解散小组时 | 将小组状态变为 dissolved，并使创建者和成员均无法在普通页面查看该小组。 |
| GROUP-STATE-006 | 小组状态为 archived 或 dissolved 时 | 系统不得再允许其变回 upcoming 或 active。 |

### 2.4 成员关系状态

| 状态 | 含义 | 是否可操作小组 | 是否计入统计 | 普通页面是否可见 |
|---|---|---|---|---|
| active | 当前有效成员关系 | 是，受小组状态限制 | 是 | 是 |
| exited | 成员主动退出 | 否 | 否 | 否 |
| removed | 创建者移除成员 | 否 | 否 | 否 |

状态变化规则：

| 规则编号 | 当……时 | 系统应…… |
|---|---|---|
| MEMBER-STATE-001 | 用户成功加入小组时 | 创建 active 成员关系。 |
| MEMBER-STATE-002 | active 成员主动退出小组并确认时 | 将成员关系变为 exited。 |
| MEMBER-STATE-003 | 创建者移除 active 成员并确认时 | 将该成员关系变为 removed。 |
| MEMBER-STATE-004 | 成员关系为 exited 或 removed 时 | 该成员不得继续提交目标、打卡或补卡。 |
| MEMBER-STATE-005 | 成员关系为 exited 或 removed 时 | 该成员历史数据物理保留，但普通页面不可见，且不计入统计；exited 成员重新加入后的允许补卡例外，以补卡规则和当前统计口径为准。 |

### 2.5 目标配置状态

目标配置状态作用于某成员在某小组某自然月内的整组目标配置，不作用于单个目标类型。

| 状态 | 含义 | 是否可新增目标类型 | 是否可修改目标配置 | 是否作为统计目标配置 |
|---|---|---|---|---|
| unset | 该成员在该小组该自然月尚未保存任何目标配置 | 是 | 否 | 否 |
| set | 该成员在该小组该自然月已保存整组目标配置，且小组仍处于 upcoming | 是 | 是 | 是，小组进入 active 后参与统计 |
| locked | 该成员在该小组该自然月的整组目标配置已锁定 | 否 | 否 | 是 |

状态变化规则：

| 规则编号 | 当……时 | 系统应…… |
|---|---|---|
| TARGET-STATE-001 | 成员在某小组某自然月尚未保存任何目标配置时 | 该成员-小组-自然月的目标配置状态为 unset。 |
| TARGET-STATE-002 | 小组状态为 upcoming，成员保存任意目标配置成功时 | 该成员-小组-自然月的目标配置状态为 set，且允许在小组状态变为 active 之前继续修改整组目标配置。 |
| TARGET-STATE-003 | 小组状态为 active，成员首次保存目标配置成功时 | 该成员-小组-自然月的目标配置状态应变为 locked。 |
| TARGET-STATE-004 | 小组从 upcoming 变为 active 时 | 已保存目标配置的成员-小组-自然月目标配置状态应变为 locked。 |
| TARGET-STATE-005 | 小组状态为 archived 或 dissolved 时 | 整组目标配置不得新增、修改或删除。 |
| TARGET-STATE-006 | 成员未设置任何目标时 | 个人综合进度显示“未设置目标”，不得按 0% 参与综合进度计算。 |
| TARGET-STATE-007 | 判断目标配置状态时 | 不得按单个目标类型分别维护 unset / set / locked。 |
| TARGET-STATE-008 | exited 成员重新加入同一小组且仍处于同一自然月时 | 不重置该成员-小组-自然月的目标配置状态；如原目标配置状态为 locked，仍不可修改；如仍为 unset，则按当前小组状态规则处理首次保存；目标进度、完成状态和完成日期仅按重新加入后的 active 参与区间内有效记录和允许补交的退出期间昨日或前天补卡记录重新计算；退出前已经形成的完成状态和完成日期不继承到重新加入后的当前统计。 |

### 2.6 打卡记录状态

为避免与“不得随意删除核心业务数据”产生歧义，本 Spec 使用 invalidated 表示记录作废，不使用物理删除语义。打卡记录状态只判断记录本身是否具备统计资格；最终是否计入某个统计视图，还必须同时满足成员关系、小组状态、统计周期、参与区间和可见性规则。

| 状态 | 含义 | 是否计入有效打卡上限 | 是否具备统计资格 | 是否普通可见 |
|---|---|---|---|---|
| valid | 当前有效记录 | 是 | 是，仍需满足成员关系、小组状态、统计周期、参与区间和可见性规则 | 是 |
| edited | 已被提交者在允许时间内修改过的有效记录 | 是 | 是，仍需满足成员关系、小组状态、统计周期、参与区间和可见性规则 | 是，并应展示修改后内容 |
| invalidated | 已作废但物理保留的记录 | 否 | 否 | 默认不在普通列表展示 |

状态变化规则：

| 规则编号 | 当……时 | 系统应…… |
|---|---|---|
| CHECKIN-STATE-001 | 成员成功提交当天打卡时 | 创建 valid 打卡记录。 |
| CHECKIN-STATE-002 | 成员成功提交补卡时 | 创建 valid 打卡记录，并标记为补卡。 |
| CHECKIN-STATE-003 | 成员在允许范围内修改打卡或补卡内容成功时 | 将记录状态标记为 edited，并保留可审计的修改信息。 |
| CHECKIN-STATE-004 | 记录本身被判定为作废时 | 应将记录标记为 invalidated，而不是物理删除；仅因成员退出、被移除或统计范围变化导致不进入当前统计时，不应把 valid 或 edited 记录改为 invalidated。 |
| CHECKIN-STATE-005 | 校验同一成员、同一小组、同一运动日期 5 次有效打卡上限时 | 只统计物理保留的 valid 和 edited 记录；不因退出、移除或统计排除而释放次数。 |

### 2.7 权限与可见性总则

| 规则编号 | 当……时 | 系统应…… |
|---|---|---|
| AUTH-001 | 用户未登录时 | 禁止访问需要用户身份的页面和操作，并提示先完成登录。 |
| AUTH-002 | 用户不是某小组 active 成员时 | 禁止查看该小组普通页面数据；创建者身份必须绑定 active 成员关系。 |
| AUTH-003 | 成员查看小组信息时 | 允许查看其他 active 成员的昵称、目标、打卡状态、运动照片、备注运动状态和统计结果。 |
| AUTH-004 | 成员尝试修改他人目标或打卡记录时 | 系统应阻止操作。 |
| AUTH-005 | 创建者查看小组信息时 | 创建者必须同时是该小组 active 成员；允许查看 active 成员的目标、打卡状态、运动照片、备注运动状态和统计结果。 |
| AUTH-006 | 创建者尝试修改或移除其他成员目标、打卡记录时 | 系统应阻止操作。 |
| AUTH-007 | 创建者执行移除成员、转让创建者、退出小组或解散小组时 | 系统应要求二次确认，并记录操作。 |
| AUTH-008 | 小组为 archived 状态时 | 仅归档快照生成时成员关系为 active 的成员可只读查看归档内容。 |
| AUTH-009 | 小组为 dissolved 状态时 | 创建者和成员均不可在普通页面查看该小组，仅后台审计或争议追溯可查。 |
| AUTH-010 | 成员关系为 exited 或 removed 时 | 该成员在该小组的历史数据普通页面不可见，且不计入统计；exited 成员重新加入后的允许补卡例外，以补卡规则和当前统计口径为准。 |

### 2.8 数据保留与审计总则

| 规则编号 | 规则 |
|---|---|
| DATA-001 | 核心业务数据不得随意物理删除。 |
| DATA-002 | 用户退出、成员被移除、小组被解散后，相关历史数据应物理保留。 |
| DATA-003 | 普通页面不可见的数据不得计入当前小组统计；退出成员重新加入后的允许补卡例外，以补卡规则和当前统计口径为准。 |
| DATA-004 | 创建小组、加入小组、退出小组、移除成员、解散小组、提交表单、修改记录、状态变化和错误应可追踪。 |
| DATA-005 | 敏感操作应保留操作人、操作对象、操作时间和操作结果。 |
| DATA-006 | invalidated 记录应保留作废前必要信息，用于审计和问题追溯。 |
| DATA-007 | 打卡和补卡上传的运动照片必须使用云存储，不得放入小程序本地代码包。 |
| DATA-008 | 运动照片和备注运动状态属于对应打卡或补卡记录的一部分，其访问权限和可见性跟随对应记录。 |
| DATA-009 | 运动照片仅支持静态图片，不支持视频、GIF 动图或非图片文件。 |
| DATA-010 | 用户加入小组或首次上传运动照片前，应明确提示运动照片会对同组 active 成员可见。 |
| DATA-011 | 运动照片加载失败时，应展示失败占位和重试入口，不得影响同一记录中运动数值和备注运动状态的展示。 |
| DATA-012 | 小组解散后，创建者和成员均不可在普通页面查看该小组；历史数据仅后台审计或争议追溯可查。 |

|---|---|---|
| FORBID-001 | 小组状态不是 active 时 | 禁止提交打卡和补卡。 |
| FORBID-002 | 小组状态为 archived 时 | 禁止修改昵称、目标、打卡或补卡。 |
| FORBID-003 | 小组状态为 dissolved 时 | 禁止创建者和成员的任何普通页面访问和业务操作。 |

| FORBID-006 | 用户尝试触发 v1.0 不包含的功能时 | 系统不应提供入口；如因历史入口进入，应展示功能不可用提示。 |

### 2.10 v1.0 明确不实现的全局能力

以下能力不得进入 v1.0 Spec 的功能规格，也不得在后续 Technical Design 或 Tasks 中实现：

- 点赞、评论、私信、提醒对方打卡等复杂社交。
- 排行榜。
- 商城。
- AI 推荐。
- 运动建议。
- 动作指导。
- 接入微信运动步数或外部设备数据。
- 通知功能，包括订阅消息、打卡提醒、补卡提醒、小组活动开始或结束提醒。
- 一点币结算、奖惩、统计、充值、目标完成返还或未完成扣除。
- 创建者设置小组可搜索或不可搜索。
- 小程序内搜索公开小组。
- 自定义运动目标类型。

## 3. 功能规格：小组创建与管理

### 3.1 用户故事

作为小组创建者，我希望可以快速创建本月或下月运动小组，并获得邀请成员加入的方式，以便在微信群中发起一个月度运动打卡活动。

作为小组创建者，我希望可以修改小组名称、分享小组、移除成员、转让创建者身份、在完成转让后退出自己创建的小组或解散小组，以便在必要时维护小组秩序。

### 3.2 功能规则

| 规则编号 | 规则 |
|---|---|
| GROUP-FUNC-001 | 创建者可选择创建本月小组或下月小组。 |
| GROUP-FUNC-002 | 小组名称为创建小组必填信息。 |
| GROUP-FUNC-003 | 每个小组人数上限为 50 人。 |
| GROUP-FUNC-004 | 创建本月小组时，若当前日期位于本月生命周期内，小组初始状态为 active。 |
| GROUP-FUNC-005 | 创建下月小组时，小组初始状态为 upcoming。 |

| CHECKIN-AC-017 | 打卡提交或修改成功时 | 更新该成员目标进度、打卡状态和统计结果。 |

### 7.4 边界条件

- 打卡提交日期和运动日期必须相同。
- 当天修改以提交日期的北京时间自然日为准。
- 单条打卡记录可包含多个运动数值字段，但至少应包含一个与已设置目标相关的有效运动数据；已设置目标依赖的运动字段必须填写。
- 打卡记录在提交当天可多次修改，普通页面展示最后一次修改后的运动数值、运动照片和备注运动状态。
- 每次修改均应保留审计记录。
- 运动照片仅支持静态图片，不支持视频、GIF 动图或非图片文件。
- 任一必填运动照片上传失败时，打卡不得提交成功，应提示用户重试或重新选择照片。
- 打卡修改时支持新增、删除和替换运动照片；保存后仍必须满足 1 至 3 张限制，并以最新照片作为该记录展示内容。
- 运动照片归档后仍按归档只读规则展示，不允许新增、删除或替换。
- 加入小组页或首次上传运动照片前，应提示“加入后，你的目标、打卡照片、备注和统计将对同组有效成员可见”。

## 8. 功能规格：运动补卡

### 8.1 用户故事

作为小组成员，我希望可以补昨日或前天的运动记录，以便在忘记当天打卡时补充真实运动情况。

作为小组成员，我希望补卡记录带有明确标记，以便小组记录保持透明。

### 8.2 功能规则

| 规则编号 | 规则 |

| MAKEUP-FUNC-002 | 补卡只允许选择提交日期的昨日或前天作为运动日期。 |
| MAKEUP-FUNC-003 | exited 成员重新加入同一小组后，如提交日期的昨日或前天属于退出期间，允许按补卡规则补交；退出期间的其他日期禁止补卡。 |
| MAKEUP-FUNC-004 | 每个用户每个自然日最多发起 3 次补卡。 |
| MAKEUP-FUNC-005 | 补卡记录计入同一成员、同一小组、被补运动日期的 5 次有效打卡上限；该上限按物理有效记录计算。 |
| MAKEUP-FUNC-006 | 补卡需填写的运动数值、运动照片、备注运动状态和已设置目标依赖字段规则与打卡一致。 |
| MAKEUP-FUNC-007 | 补卡无需填写原因。 |
| MAKEUP-FUNC-008 | 补卡无需创建者审批。 |
| MAKEUP-FUNC-009 | 补卡成功后应展示补卡标记。 |
| MAKEUP-FUNC-010 | 所有 active 成员都能看到补卡标记、运动照片、备注运动状态及补卡内容。 |
| MAKEUP-FUNC-011 | 可于补卡提交当天修改运动数值、增删运动照片及修改备注运动状态。 |
| MAKEUP-FUNC-012 | 补卡修改后记录状态应为 edited，并保留审计信息。 |

### 8.3 验收标准

| 编号 | 当……时 | 系统应…… |
|---|---|---|
| MAKEUP-AC-001 | active 成员在 active 小组中进入补卡流程时 | 允许选择昨日或前天作为运动日期。 |
| MAKEUP-AC-002 | 成员选择今天作为补卡运动日期时 | 阻止提交，并提示“今日运动请使用打卡”。 |
| MAKEUP-AC-003 | 成员选择早于前天的日期补卡时 | 阻止提交，并提示“只能补昨日或前天的卡”。 |
| MAKEUP-AC-004 | exited 成员重新加入后选择退出期间的昨日或前天补卡时 | 允许按补卡规则提交，并计入重新加入后的当前统计。 |
| MAKEUP-AC-005 | exited 成员重新加入后选择退出期间早于前天的日期补卡时 | 阻止提交，并提示“只能补昨日或前天的卡”。 |
| MAKEUP-AC-006 | 成员在 upcoming、archived 或 dissolved 小组中提交补卡时 | 阻止提交，并提示对应小组状态不可补卡。 |
| MAKEUP-AC-007 | exited 或 removed 成员提交补卡时 | 阻止提交，并提示“你已不属于该小组”。 |
| MAKEUP-AC-008 | 成员在同一提交日期已发起 3 次补卡时 | 阻止继续补卡，并提示“今日补卡次数已用完”。 |
| MAKEUP-AC-009 | 同一成员、同一小组、被补运动日期已有 5 条物理保留的 valid 或 edited 记录时 | 阻止补卡，并提示“该运动日期已达到 5 次有效打卡上限”。 |
| MAKEUP-AC-010 | 成员提交的补卡运动数据无效时 | 阻止提交，并提示“请输入有效运动数据”。 |
| MAKEUP-AC-011 | 成员未上传运动照片、少于 1 张或超过 3 张时 | 阻止提交，并提示“请上传 1 至 3 张运动照片”。 |
| MAKEUP-AC-012 | 成员填写备注运动状态超过 100 字时 | 阻止提交，并提示“备注最多 100 字”。 |
| MAKEUP-AC-013 | 成员补卡成功时 | 创建 valid 打卡记录，标记为补卡，并保存被补运动日期、运动数值、运动照片和备注运动状态。 |
| MAKEUP-AC-014 | 成员在补卡提交当天修改自己的补卡记录时 | 允许修改运动数值、增删运动照片及修改备注运动状态，并将记录状态标记为 edited。 |
| MAKEUP-AC-015 | 成员修改后运动照片少于 1 张或超过 3 张时 | 阻止保存，并提示“请上传 1 至 3 张运动照片”。 |
| MAKEUP-AC-016 | 成员在补卡提交日期之后修改补卡记录时 | 阻止修改，并提示“仅可在补卡当天修改”。 |
| MAKEUP-AC-017 | 补卡提交或修改成功时 | 更新被补运动日期的打卡状态、目标进度和统计结果。 |

### 8.4 边界条件

- 补卡次数限制按提交日期计算，不按被补运动日期计算。
- 补卡记录计入同一成员、同一小组、被补运动日期的 5 次有效打卡上限；5 次上限按物理有效记录计算，不因退出、移除或统计排除而释放次数。
- exited 成员重新加入后，退出期间只有提交日期的昨日或前天可补卡；退出期间其他日期不得补卡。
- 补卡无需填写原因，系统不得强制展示原因输入。
- 补卡无需审批，创建者不得拥有审批入口。
- 补卡标记应在成员可见的打卡记录中展示。
- 运动照片仅支持静态图片，不支持视频、GIF 动图或非图片文件。
- 任一必填运动照片上传失败时，补卡不得提交成功，应提示用户重试或重新选择照片。
- 补卡修改时支持新增、删除和替换运动照片；保存后仍必须满足 1 至 3 张限制，并以最新照片作为该记录展示内容。
- 运动照片归档后仍按归档只读规则展示，不允许新增、删除或替换。
- 加入小组页或首次上传运动照片前，应提示“加入后，你的目标、打卡照片、备注和统计将对同组有效成员可见”。

- 归档页中的运动照片仅可只读查看，不允许新增、删除或替换。

## 10. 功能规格：统计与复盘

作为小组创建者或成员，我希望可以查看已完成月度目标的成员、完成日期和小组整体完成率，以便完成月度复盘。

| STATS-FUNC-004 | 月目标完成以所有目标均达到 100% 为准。 |

| STATS-FUNC-014 | 小组归档时应冻结成员完成状态、完成日期和小组整体完成率；归档后不因后续数据修正、成员状态变化或重新加入行为重新计算普通页面统计结果。 |

### 10.3 进度计算规则

| 目标类型 | 完成值定义 | 目标值定义 | 单项目标进度 |

| 月总运动时长 | 统计范围内，当月具备统计资格的 valid 和 edited 记录累计运动时长 | 目标小时数 | min(完成值 / 目标值, 100%) |
| 月总最低消耗热量 | 统计范围内，当月所有具备统计资格的 valid 和 edited 记录累计消耗热量 | 目标 kcal | min(完成值 / 目标值, 100%) |
| 跑步距离 | 统计范围内，当月具备统计资格的 valid 和 edited 记录累计跑步距离 | 目标 km | min(完成值 / 目标值, 100%) |
| 骑行距离 | 统计范围内，当月具备统计资格的 valid 和 edited 记录累计骑行距离 | 目标 km | min(完成值 / 目标值, 100%) |
| 三环闭合天数 | 统计范围内，当月存在至少一条具备统计资格的 valid 或 edited 记录被用户手动标记为三环闭合的运动日期数；同一运动日期最多计 1 天 | 目标天数 | min(完成值 / 目标值, 100%) |

### 10.4 验收标准

| 编号 | 当……时 | 系统应…… |
|---|---|---|
| STATS-AC-001 | 成员已设置一个或多个目标时 | 展示每个目标类型的单项目标进度。 |
| STATS-AC-002 | 单项目标完成值超过目标值时 | 单项目标进度最高显示为 100%。 |
| STATS-AC-003 | 成员已设置多个目标时 | 个人综合进度显示为所有已设置目标的单项目标进度平均值。 |
| STATS-AC-004 | 成员未设置任何目标时 | 个人综合进度显示“未设置目标”。 |
| STATS-AC-005 | 成员所有已设置目标在当前统计口径下首次均达到 100% 时 | 标记该成员完成月度目标，并记录完成日期。 |
| STATS-AC-006 | 成员部分目标达到 100%、部分目标未达到 100% 时 | 不得标记为完成月度目标。 |
| STATS-AC-007 | 成员完成月度目标时 | 完成日期按当前统计口径下最后一个未完成目标首次达到 100% 的日期计算。 |
| STATS-AC-008 | 计算小组整体完成率时 | 使用当前 active 成员作为分母，已完成月度目标的 active 成员作为分子。 |
| STATS-AC-009 | 成员 exited 或 removed 后 | 该成员不计入小组整体完成率分子或分母。 |
| STATS-AC-010 | 小组从 active 变为 archived 时 | 生成归档统计快照，快照按归档时刻的 active 成员、有效目标配置和具备统计资格的 valid / edited 打卡记录计算。 |
| STATS-AC-011 | 小组 dissolved 时 | 普通统计页面不展示该小组统计，创建者和成员均不可通过普通页面查看。 |
| STATS-AC-012 | archived 小组被查看时 | 只读展示归档统计快照，不因归档后的成员关系、目标配置或记录状态变化重算普通页面统计。 |
| STATS-AC-013 | 计算运动天数目标时 | 按同一运动日期内所有具备统计资格的 valid 和 edited 记录累计热量判断是否达到单天最低消耗热量；达标日期计 1 天。 |
| STATS-AC-014 | 计算运动次数目标时 | 按单条具备统计资格的 valid 或 edited 记录热量判断是否达到单次最低消耗热量；同一运动日期多条未达标记录不得累计折算为次数。 |
| STATS-AC-015 | 计算三环闭合天数目标时 | 同一运动日期存在至少一条具备统计资格的 valid 或 edited 记录标记为三环闭合时，该日期计 1 天；同一运动日期最多计 1 天。 |
| STATS-AC-016 | exited 成员重新加入同一小组后查看当前目标进度时 | 沿用原目标配置作为当月目标，仅使用重新加入后的 active 参与区间内具备统计资格的 valid 和 edited 记录，以及允许补交的退出期间昨日或前天补卡记录计算当前进度。 |
| STATS-AC-017 | exited 成员重新加入同一小组后重新达到月目标完成条件时 | 按重新加入后的当前统计口径重新标记完成状态并记录完成日期；退出前已经形成的完成状态和完成日期不继承到重新加入后的当前统计。 |
| STATS-AC-018 | archived 小组被查看时 | 展示归档时冻结的完成状态、完成日期和小组整体完成率，不因后续变化重新计算。 |

### 10.5 边界条件

- 统计只使用具备统计资格的 valid 和 edited 打卡记录。
- invalidated 记录不计入任何进度和完成率。
- 补卡记录按其运动日期计入目标进度。
- 一点币不参与任何统计计算。
- 小组整体完成率在分母为 0 时显示“暂无有效成员”，不得显示为 0% 或 100%。
- 运动天数和运动次数使用不同的热量达标口径：运动天数按同一运动日期累计热量判断，运动次数按单条记录热量判断。
- 统计范围默认指当前 active 参与区间；exited 成员重新加入后，退出前记录不进入当前目标进度，退出期间仅允许昨日或前天的补卡记录按补卡规则进入当前目标进度；完成状态和完成日期也按该当前统计口径重新计算。
- 归档统计快照用于普通页面只读展示；后台审计或争议追溯可按原始数据另行查询。
- 归档快照应冻结完成状态、完成日期和小组整体完成率。

## 11. 功能规格：退出、移除成员、转让创建者、解散小组

### 11.1 用户故事

作为小组成员，我希望可以主动退出小组，以便停止参与该小组的后续打卡和统计。

作为小组创建者，我希望可以转让创建者身份、完成转让后退出自己创建的小组、移除成员或解散小组，以便处理成员变动、完成退出前的身份交接或终止小组活动。

### 11.2 功能规则

| 规则编号 | 规则 |
|---|---|
| RELATION-FUNC-001 | active 成员可以主动退出小组。 |
| RELATION-FUNC-002 | 成员退出小组前，系统应要求二次确认。 |
| RELATION-FUNC-003 | 成员退出后，成员关系状态变为 exited。 |
| RELATION-FUNC-004 | 创建者可以移除 active 成员。 |
| RELATION-FUNC-005 | 创建者移除成员前，系统应要求二次确认。 |
| RELATION-FUNC-006 | 成员被移除后，成员关系状态变为 removed。 |
| RELATION-FUNC-007 | 创建者退出自己创建的小组前，必须先将创建者身份转让给其他 active 成员。 |
| RELATION-FUNC-008 | 创建者可以解散小组。 |
| RELATION-FUNC-009 | 创建者解散小组前，系统应要求二次确认。 |
| RELATION-FUNC-010 | 小组被解散后，小组状态变为 dissolved，创建者和成员均不可在普通页面查看该小组。 |
| RELATION-FUNC-011 | 退出、移除、解散后的相关历史数据物理保留。 |
| RELATION-FUNC-012 | 退出、移除、解散后的相关数据普通页面不可见，且不计入统计；exited 成员重新加入后的允许补卡例外，以补卡规则和当前统计口径为准。 |
| RELATION-FUNC-013 | 退出、移除、转让创建者、解散操作应记录审计信息。 |

### 11.3 验收标准

| 编号 | 当……时 | 系统应…… |
|---|---|---|
| RELATION-AC-001 | active 成员发起退出小组时 | 展示二次确认。 |
| RELATION-AC-002 | active 成员确认退出小组时 | 将成员关系状态改为 exited。 |
| RELATION-AC-003 | 成员退出小组后 | 禁止其继续提交目标、打卡或补卡。 |
| RELATION-AC-004 | 创建者移除 active 成员时 | 展示二次确认。 |
| RELATION-AC-005 | 创建者确认移除成员时 | 将该成员关系状态改为 removed。 |
| RELATION-AC-006 | 成员被移除后 | 该成员不再出现在普通成员列表和统计中。 |
| RELATION-AC-007 | 非创建者尝试移除成员时 | 阻止操作，并提示“无权限操作”。 |
| RELATION-AC-008 | 创建者发起退出自己创建的小组但尚未转让创建者身份时 | 阻止退出，并提示“请先转让创建者身份”。 |
| RELATION-AC-009 | 创建者已将创建者身份转让给其他 active 成员后发起退出时 | 展示二次确认。 |
| RELATION-AC-010 | 创建者完成转让并确认退出时 | 将原创建者成员关系状态改为 exited。 |
| RELATION-AC-011 | 创建者发起解散小组时 | 展示二次确认。 |
| RELATION-AC-012 | 创建者确认解散小组时 | 将小组状态改为 dissolved。 |
| RELATION-AC-013 | 小组解散后 | 创建者和成员均不可在普通页面查看该小组，且不计入统计；历史数据仅后台审计或争议追溯可查。 |
| RELATION-AC-014 | 退出、移除或解散发生后 | 相关历史数据物理保留。 |
| RELATION-AC-015 | 退出、移除、转让创建者或解散成功后 | 记录操作人、操作对象、操作时间和操作结果。 |

### 11.4 边界条件

- 创建者允许退出自己创建的小组，但退出前必须先将创建者身份转让给其他 active 成员。
- exited 成员可通过有效入口重新加入同一小组；重新加入后生成新的 active 参与区间，退出前历史数据不补回当前统计，且不重置该成员-小组-自然月的目标配置状态；如原目标配置已存在，重新加入后沿用原目标配置作为当月目标。
- exited 成员重新加入同一小组后，当前月完成状态和完成日期按重新加入后的当前统计口径重新计算；退出前已经形成的完成状态和完成日期不继承到重新加入后的当前统计，仅作为历史数据保留。
- removed 成员在 v1.0.0 MVP 中不可通过原邀请码、分享卡片或其他有效入口重新加入同一小组。
- 退出前数据不补回当前统计；exited 成员重新加入后，从新的 active 参与区间开始参与后续统计；如昨日或前天属于退出期间，允许按补卡规则补交并计入当前统计，退出期间的其他日期禁止补卡。
- 被移除成员的历史数据物理保留，但普通页面不可见且不计入统计；v1.0.0 不提供 removed 成员恢复为 active 的路径。
- 解散小组是不可逆操作；解散后创建者和成员均不可在普通页面查看该小组，仅后台审计或争议追溯可查。

- 首页 / 小组入口页应区分可进入小组、归档小组和不可见小组。
- 创建小组页应展示本月小组和下月小组的生命周期说明。
- 加入小组页应在用户确认前展示小组名称、生命周期、成员数和照片/目标/备注/统计可见性提示。
- 小组详情页在 archived 状态下必须隐藏打卡、补卡、目标修改、昵称修改、移除成员和解散入口。
- 目标设置页应在目标锁定时展示锁定原因。
- 打卡页应展示同一成员、同一小组、同一运动日期 5 次有效打卡限制状态；该状态按物理保留的 valid / edited 记录计算，不受当前统计资格影响。
- 打卡页应展示运动照片上传入口、1 至 3 张数量限制、备注 100 字限制和首次上传照片前的可见性提示。
- 补卡页应展示每个用户每个自然日 3 次补卡限制状态。
- 补卡页应展示被补运动日期的 5 次有效打卡限制状态；该状态按物理保留的 valid / edited 记录计算，不受当前统计资格影响。
- 补卡页应展示与打卡一致的运动照片上传入口、1 至 3 张数量限制、备注 100 字限制和首次上传照片前的可见性提示。
- 统计复盘页应展示“未设置目标”“暂无有效成员”等空状态。
