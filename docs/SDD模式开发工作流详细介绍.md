## 01. 什么是 SDD：规格驱动开发
先把“要实现什么”定义为可执行规格，再让人或 AI 按规格实现。
SDD 是 Spec-Driven Development 的简称，通常可译为“规格驱动开发”或“规格先行开发”。它的核心不
是写更多文档，而是让文档成为开发过程的控制系统：需求、技术方案、任务拆解、代码实现、测试验收都围
绕同一套规格推进。
在 AI 编码场景里，SDD 的价值更明显。AI 工具擅长生成代码，但如果输入只有一句模糊想法，它会自行补全
大量假设；这些假设可能和真实需求不一致。SDD 通过 PRD、Spec、Design、Tasks 等文档，把模糊空间压
缩到可验证范围内。
核心原则
先写清楚“业务规则”和“验收标准”，再进入编码。
任何没有验收标准的功能，都不应该直接开发。
关键词 含义 在开发中的作用
PRD 产品需求文档 回答为什么做、给谁用、做哪些功能
Spec 功能规格说明 把需求写成可开发、可测试的规则
Design 技术设计方案 回答系统怎么实现、数据怎么存、接口
怎么设计
Tasks 任务拆解清单 把开发拆成可单独完成和测试的小任务
Test Checklist 测试检查表 用验收标准判断功能是否完成
Change Log 变更记录 记录需求调整，避免文档和代码脱节

## 05. 阶段 0：Project Principles / Constitution
Project Principles 也可以叫项目约束、开发原则或 Constitution。它不是写具体功能，而是规定项目长期必
须遵守的边界。
原则类别 应写清楚的问题 示例
产品边界 MVP 先做什么，不做什么 先做打卡、记录、排行榜；暂不做商
城、私信、复杂社区
技术栈 使用什么框架和服务 微信小程序原生框架 + 云开发
性能约束 包体积、加载速度、资源管理 主包控制在 2MB 以内，大图放云存储
安全约束 用户数据和权限怎么处理 用户只能读写自己的打卡记录，排行榜
仅读取聚合结果
开发规则 AI 修改代码的范围 每次只实现一个任务，不允许顺手重构
无关文件
验收规则 完成的定义是什么 必须通过测试清单后才能进入下一任务

# 00-project-principles.md 示例

## 项目目标
做一个面向 20 人左右运动群的小程序，用于每月运动打卡、统计、复盘和互相监督。

## 开发原则
1. 先做 MVP，不做复杂社交、商城、AI 推荐等非核心功能。
2. 使用微信小程序原生框架 + 云开发。
3. 图片、头像、运动截图等大文件必须使用云存储，不放入本地代码包。
4. 小程序主包体积必须控制在 2MB 以内。
5. 所有功能必须先有需求说明和验收标准，再进入开发。
6. AI 编码工具每次只允许完成一个明确任务。
7. 需求变更必须先更新 Spec，再更新代码。

## 06. 阶段 1：PRD 产品需求文档
PRD 用来回答“为什么做、给谁用、做什么、做到什么程度”。对 SDD 来说，PRD 不需要追求长，但必须明
确产品目标、用户角色、核心流程和 MVP 范围。
模块 说明 示例问题
背景与目标 说明项目存在的原因和业务目标 为什么要做这个小程序？想解决什么问
题？
用户角色 定义谁会使用产品 群主、普通参与者、未加入活动用户分
别能做什么？
核心场景 描述用户何时、为何、如何使用 每天运动后怎么打卡？月底怎么复盘？
功能范围 明确 MVP 包含和不包含的功能 是否支持补卡？是否支持上传截图？
页面结构 列出主要页面 首页、打卡页、排行榜页、我的页面
业务规则 写出关键限制 一天只能打卡一次；排行榜按本月累计
天数排序
验收标准 初步定义完成标准 用户打卡成功后，首页状态应立即更新
MVP 范围建议
运动打卡小程序第一版建议只做：用户登录、加入活动、每日打卡、打卡记录、排行榜、我的页面。复杂勋章、
私聊、AI 推荐、积分商城都应延后。
给 AI 的 PRD 生成提示词：
请基于以下产品想法，生成一份微信小程序 MVP 版本 PRD。
要求：
1. 只输出 PRD，不要写代码。
2. 功能范围控制在最小可上线版本。
3. 明确用户角色、核心流程、页面结构、业务规则、验收标准。
4. 技术背景：微信小程序原生框架 + 云开发。
5. 注意主包体积不得超过 2MB。
产品想法：
我想做一个运动打卡小程序，供我和朋友们每月记录运动情况，互相监督，坚持每天一点点运动。

## 07. 阶段 2：Spec 功能规格说明
Spec 是 SDD 的核心。它要把 PRD 中的产品语言转换成开发可执行、测试可验证的规格。推荐使用“用户故
事 + 验收标准 + 边界条件”的结构。
PRD 表达 Spec 表达
用户可以每日运动打卡。 当用户已登录且今日未打卡时，系统应允许用户提交运动类
型、运动时长和备注。
排行榜展示运动情况。 排行榜应按本月累计打卡天数降序排列；打卡天数相同时，按
总运动时长降序排列。
支持上传运动截图。 当用户选择上传图片时，系统应将图片上传至云存储，并在
checkins 集合中保存 fileID。
防止重复打卡。 当系统检测到用户当天已有打卡记录时，应禁止再次提交，并
提示“今日已完成打卡”。

# 02-spec.md 示例片段

## 功能：每日打卡

### 用户故事
作为活动参与者，我希望每天提交一次运动记录，以便记录自己的运动坚持情况。

### 验收标准
1. 当用户未登录时，系统应禁止进入打卡提交页。
2. 当用户已登录且今日未打卡时，系统应允许用户提交运动类型、运动时长和备注。
3. 当运动时长为空或小于等于 0 时，系统应阻止提交并提示用户修改。
4. 当用户提交成功后，系统应创建一条当天打卡记录，并更新首页今日状态。
5. 当用户今日已打卡时，系统应禁止重复提交，并提示“今日已完成打卡”。
6. 当网络异常时，系统应提示“提交失败，请稍后重试”。

### 边界条件
- 每个自然日按用户本地日期计算。
- 同一用户、同一活动、同一日期只能存在一条有效打卡记录。
- 图片不是必填项；备注不是必填项。
给 AI 的 Spec 生成提示词：
请根据 docs/01-prd.md 生成 docs/02-spec.md。
要求：
1. 不要写代码。
2. 把每个功能拆成用户故事。
3. 每个用户故事必须包含验收标准。
4. 验收标准使用“当……时，系统应……”格式。
5. 明确正常流程、异常流程、边界条件。
6. 如果 PRD 中有模糊点，请列出问题，不要自行假设。

## 08. 阶段 3：Technical Design 技术设计
Technical Design 用来回答“系统怎么实现”。它不是代码，而是结构图、页面划分、数据库设计、接口设
计、权限规则和风险方案。
设计项 需要定义的内容 示例
页面结构 页面路径、页面职责、跳转关系 pages/index、pages/checkin、
pages/ranking、pages/me
组件结构 是否抽公共组件 checkin-card、ranking-item、empty-state
数据库集合 集合名称、字段、索引、关系 users、activities、checkins
云函数 函数名、输入、输出、错误处理 login、createCheckin、getRanking
权限规则 谁能读写哪些数据 用户只能创建自己的 checkin 记录
资源策略 图片与素材放在哪里 大图放云存储，本地只放压缩图标
性能策略 如何控制包体积和请求次数 分页加载、避免大型 npm 包

## 09. 阶段 4：Tasks 开发任务拆解
Tasks 是从技术方案到实际执行之间的桥。一个好任务必须足够小，可以独立开发、独立测试、独立回滚。
好任务 坏任务
T004 实现打卡表单页面 实现整个运动打卡系统
T006 实现 createCheckin 云函数 完成所有云函数和数据库
T009 实现排行榜列表渲染 实现全部页面并优化 UI
T012 增加重复打卡校验 修复所有问题

## T004 实现打卡表单页面
目标：
用户可以填写运动类型、运动时长、备注，并点击提交。
涉及文件：

- miniprogram/pages/checkin/checkin.wxss
- miniprogram/pages/checkin/checkin.js
- miniprogram/app.json

- T002 用户登录已完成
开发步骤：
1. 创建 checkin 页面
2. 添加运动类型选择
3. 添加运动时长输入
4. 添加备注输入
5. 添加提交按钮
6. 做基础表单校验

1. 运动时长为空时不能提交。

请根据 docs/03-technical-design.md 生成 docs/04-tasks.md。

## 10. 阶段 5：Implement 逐任务编码
进入 Implement 阶段后，AI 编码工具只能根据已确认的任务清单执行。不要让它一次性实现多个页面、多个
云函数和多个业务逻辑。
执行纪律
每次只实现一个任务。
只修改任务涉及文件。
如果需要修改额外文件，必须先说明原因。
完成后必须列出改动文件和测试方法。
逐任务编码提示词：
请只实现 docs/04-tasks.md 中的 T004。
要求：
1. 只修改 T004 涉及的文件。
2. 不要实现其他任务。
3. 如果发现需要修改其他文件，先说明原因。
4. 完成后列出修改了哪些文件。
5. 完成后说明如何在微信开发者工具中测试。
6. 不要引入大型 npm 包，不要添加本地大图片或无关资源。
执行前检查 通过标准
是否有明确任务编号？ 例如 T004，而不是“继续完善项目”
是否有限定文件范围？ 明确哪些文件可改，哪些不能改
是否有验收标准？ 可以在开发者工具中逐条验证
是否限制 AI 自由发挥？ 禁止实现任务外功能
是否考虑小程序约束？ 包体积、云存储、API 支持情况已说明

## 11. 阶段 6：Test 验收与回归测试
SDD 的测试不是等所有功能完成后再做，而是每个任务完成后立即验证。测试清单应直接来自 Spec 的验收标
准。
测试类型 测试内容 示例
功能测试 功能是否按规格运行 未登录不能打卡；已打卡不能重复提交
边界测试 异常值和极端情况 运动时长为 0、为空、非数字
数据测试 数据库是否写入正确字段 checkins 是否保存 openid、date、
duration
权限测试 用户是否只能操作自己的数据 用户 A 不能修改用户 B 的打卡记录
回归测试 新增功能是否破坏旧功能 打卡后排行榜是否仍正常排序
发布测试 能否预览、上传、发布 主包是否小于 2MB

## 12. 阶段 7：Change Log 变更管理
需求一定会变化。SDD 不是禁止变化，而是要求变化被记录、被评估、被同步。Change Log 的作用是避
免“代码改了，但文档没改；文档改了，但任务没改”的失控状态。
变更类型 处理方式
新增功能 先更新 PRD 和 Spec，再补充 Design 和 Tasks
修改业务规则 优先更新 Spec 的验收标准，再检查相关测试清单
技术方案调整 更新 Technical Design，并标记受影响任务
修复缺陷 记录缺陷来源、修复文件、回归测试结果
删除功能 标记删除原因，并检查是否有遗留代码或数据库字段

## 13. 适用于微信小程序的特别注意事项

程序运行环境。

## 14. AI 编码工具提示词模板
下面这些模板可以直接复制到 AI 编码工具中使用。关键不是模板本身，而是每次都明确：参考文档、执行范
围、禁止事项、验收方式。
模板 A：让 AI 阅读现有规格，不写代码
请阅读 docs/00-project-principles.md、docs/01-prd.md、docs/02-spec.md。
请只输出你对项目目标、功能范围、关键约束和未明确问题的理解。
不要写代码，不要修改文件。
模板 B：让 AI 检查 Spec 是否完整
请检查 docs/02-spec.md 是否适合进入开发。
重点检查：
1. 是否每个功能都有用户故事。
2. 是否每个用户故事都有验收标准。
3. 是否覆盖正常流程、异常流程、边界条件。
4. 是否存在模糊表述，例如“优化”“完善”“友好展示”。
5. 请输出问题清单和修改建议，不要写代码。
模板 C：让 AI 按任务实现

要求：
1. 参考 docs/00-project-principles.md、docs/02-spec.md、docs/03-technical-design.md。
2. 只修改该任务涉及文件。
3. 不要实现任务外功能。
4. 不要引入大型依赖，不要添加本地大图片。
5. 完成后输出：修改文件列表、实现说明、测试步骤、风险提示。
模板 D：让 AI 修复错误
当前任务：T{任务编号}
错误现象：{粘贴错误信息或截图文字}
相关文件：{列出或粘贴相关文件}
请分析错误原因，并给出最小修复方案。
要求：
1. 只修复与该错误直接相关的问题。
2. 不要顺手重构无关代码。

## 15. 常见错误与纠偏方式
常见错误 后果 纠偏方式
直接让 AI 生成完整项目 代码量暴涨，错误难定位 先生成 PRD/Spec/Design/Tasks，再逐
任务实现
需求只有一句话 AI 自行脑补功能和规则 补充用户角色、业务规则、验收标准
任务过大 一次改动过多，无法测试 拆成单页面、单云函数、单校验规则
没有变更记录 文档和代码不一致 每次需求调整写入 change-log
本地放大资源 包体积超限 大图、截图、头像全部云存储
忽略异常流程 上线后用户一操作就出错 Spec 中强制写异常流程和边界条件
只看功能能跑 数据、权限、发布环节埋雷 增加数据测试、权限测试、发布测试
最重要的纠偏动作
任何时候项目开始失控，都退回到 Spec：重新确认功能边界、业务规则和验收标准。不要在混乱状态下继续让
AI 叠代码。

最后的执行建议
对 AI 辅助开发来说，SDD 的本质是“用规格约束生成”。你不需要一开始就把文档写成企业级规范，但必须
做到三件事：
1. 把需求写成明确规则，而不是停留在想法层。
2. 把规则写成可以测试的验收标准。
3. 把开发拆成小任务，每次只让 AI 完成一个任务。
一句话总结
SDD 不是为了增加文档负担，而是为了降低 AI 编码的不确定性：先定规格，再写代码；先能验收，再谈完成。
